```
Fintech Wallet Backend

Fintech Wallet Backend is a learning-oriented payment system inspired by modern digital wallets and payment applications. The project demonstrates how authentication, wallet management, money transfers, transaction history, and database consistency work behind the scenes. It combines NoSQL-style document storage for user information and SQL transactions for financial operations, helping developers understand why real-world payment systems rarely rely on a single storage strategy.
```

Why I Built This Project

Modern payment applications such as digital wallets, banking apps, and fintech platforms need much more than simple CRUD operations.

They must authenticate users securely, maintain wallet balances accurately, prevent money from disappearing during failures, and preserve a complete transaction history for auditing.

This project was built to understand the core engineering principles behind these systems and implement them in a simplified but realistic backend using FastAPI, SQLite, JSON, and JWT authentication.

   │

   │      └── SQLite
   │

The application follows a layered service architecture. Client requests are received by FastAPI endpoints, validated using Pydantic schemas, processed through business services, and persisted into storage systems. Authentication concerns are separated from wallet operations, and financial transactions are isolated inside SQL transactions to maintain consistency and reliability.

```
Why SQL + NoSQL?

JSON (NoSQL)

User information is naturally flexible. Different users may have different profile settings, preferences, addresses, and metadata. Document-based storage makes it easy to evolve user information without requiring frequent schema migrations.

``` 
{
  "id": "USR-123",
  "name": "Sarthak",
  "email": "sarthak@gmail.com",
  "wallet_id": "WAL-123"
}
```

Flexible user information, preferences, profile data, settings, and documents.

SQLite (SQL)

Financial operations require guarantees. Wallet balances and transactions must remain consistent even if the server crashes in the middle of a transfer. SQL databases provide transactions, rollback mechanisms, relational integrity, and ACID properties, making them ideal for monetary data.

``` 
wallets
transactions
audit records
balances
```

Financial data requires consistency, transactions, rollback, and relational integrity.
```

```
Money Transfer Flow

``` 
BEGIN
    ↓
Validate Sender
    ↓
Validate Receiver
    ↓
Check Balance
    ↓
Deduct Sender
    ↓
Credit Receiver
    ↓
Insert Transaction Record
    ↓
COMMIT

Every money transfer is executed inside a single database transaction. If any step fails, the database performs a rollback and returns to its previous state. This prevents situations where money is deducted from one wallet but never credited to another wallet. This concept of atomicity is one of the most important principles in financial systems.

ROLLBACK

Authentication Flow 

Register
    ↓
Hash Password
    ↓
Store User
    ↓
Login
    ↓
Verify Credentials
    ↓
Generate JWT Token
    ↓
Access Protected APIs

Passwords are never stored in plain text. They are securely hashed before storage. After login, a JWT access token is generated and used to authorize protected wallet operations such as deposits, withdrawals, and money transfers.

Key Engineering Concepts

JWT Authentication

Secure stateless authentication using access tokens and protected endpoints.

Service Layer Architecture

Business logic is separated from API routes, making the system easier to maintain and extend.

ACID Transactions

Financial operations are executed atomically using BEGIN, COMMIT, and ROLLBACK.

Hybrid Storage Strategy

Combining JSON-based documents with relational databases demonstrates practical trade-offs between flexibility and consistency.
